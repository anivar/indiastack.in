Code first. Policy later. How private infrastructure becomes public mandate.

PRware: The Brand Without a Codebase

"India Stack" is PRware, a marketing term by iSPIRT that brands separate closed systems as unified "open" infrastructure. There is no entity called "India Stack." Aadhaar is owned by UIDAI. UPI by NPCI. ABDM by NHA. DigiLocker by MeitY. All closed source. All separately governed.

The branding came first. The code exists in silos. Private entities build it. Government adopts, endorses, and mandates it. By the time Parliament debates, the infrastructure is already deployed. The architecture is already the policy.

The Sequence

This is not how public infrastructure is supposed to work. But this is how it actually works:

1
Code
Private sector builds it. "Volunteers" design specifications. Think tanks write frameworks. Industry associations draft protocols. No public tender. No parliamentary approval. No citizen input.
↓
2
Deployment
Pilot programs in select states. "Innovation" narratives in friendly media. Early adopters create ecosystem. Startups build on the infrastructure. Facts on the ground established.
↓
3
Adoption Push
Government agencies told to integrate. Banks, telcos, hospitals given "guidelines." States offered funding, conditional on adoption. Network effects kick in. Opting out becomes costly.
↓
4
Political Endorsement
High-level endorsement. International forums. G20 showcase. "India's gift to the world." Success metrics announced. Critics labeled anti-national. The infrastructure is now political identity.
↓
5
Policy / Law
Legislative amendments. Years after deployment. Enabling acts that legitimize what already exists. Retrospective legal cover for fait accompli. Debate happens after the fact.
Notice the order. Code comes first. Democratic deliberation comes last. By the time there's a law, there's already an ecosystem, a narrative, political capital invested, and millions of users with no alternative.

Why This Sequence Matters

Reversibility Decreases at Each Stage

  • At Stage 1 (Code): Could have been designed differently. Open source or closed. Centralized or federated. With consent or without. All choices still open.
  • At Stage 2 (Deployment): Early adopters invested. Changing architecture means rebuilding. But still possible.
  • At Stage 3 (Adoption): Ecosystem dependent. Banks, apps, government services integrated. Switching costs high. Network effects lock in.
  • At Stage 4 (Endorsement): Political capital invested. Criticism becomes politically costly. Alternatives seen as "anti-progress."
  • At Stage 5 (Law): Legal framework built around existing architecture. Changing requires new legislation. System is now "constitutional."
Democratic input arrives when changing course is most difficult. The window for genuine choice closes before the debate begins.

Why "Public" Is Misleading

  • Not publicly owned: Code owned by private entities or foundations controlled by private interests.
  • Not publicly governed: Specifications set by industry consortiums, not elected bodies.
  • Not publicly auditable: Core systems closed source. No independent security audits published.
  • Not publicly contestable: Once mandated, alternatives become legally or practically impossible.
"Digital Public Infrastructure" means: infrastructure the public must use. Not: infrastructure the public controls.

What Would Democratic Governance Look Like?

The Sequence Should Be Reversed

  • Policy first: Parliament debates what infrastructure is needed and what values it should embody.
  • Public design: Specifications developed through open, documented process with civil society input.
  • Competitive implementation: Multiple vendors can build to spec. No single-source monopoly.
  • Gradual adoption: Voluntary before mandatory. Alternatives preserved.
  • Continuous oversight: Regular audits. Published results. Accountability for failures.

Questions That Should Be Asked Before Code

  • What problem does this solve? Who defined the problem?
  • What alternatives were considered? Why were they rejected?
  • Who benefits from this architecture? Who bears the risks?
  • What happens when it fails? Who is liable?
  • Can citizens opt out? What are the consequences of opting out?
  • Who controls the data? Under what legal framework?
  • Is the code auditable? By whom?
  • How can this be changed if it doesn't work?
These questions are never asked before the code is written. By the time they're asked, the answer is: "It's already deployed. Millions of people depend on it. We can't change it now."
