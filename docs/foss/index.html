The Observability Problem

"Open source" is about a fundamental question of power: can citizens observe and correct systems that govern their lives?

Is the source code public? No.
Can you verify the implementation? No.
Can citizens audit the code? No.
Does it meet OSI definition? No.
First, a Clarification

"India Stack" is PRware, a marketing term coined by iSPIRT. There is no entity called "India Stack." Each system is owned by a different institution: UIDAI (Aadhaar), NPCI (UPI), NHA (ABDM), MeitY (DigiLocker). All closed source. The brand is unified. The code is fragmented, proprietary, and unobservable.

First Principle

If you can't see what a system does, you can't fix it. And what you can't fix eventually owns you. Basic control theory.

In cybernetic terms, governance requires a feedback loop: citizens observe the system's behavior, compare it to desired outcomes, and correct errors. When the system is a black box, when the code is hidden, this loop breaks. Errors accumulate. Power concentrates. The governed become the controlled.

The Layers of Deception

1

2

→ Mostly OpenAPI docs, how to call an API, not how the system works
3
"You can see what it does"
→ API documentation ≠ system behavior. Implementation can differ from docs.
4
"It's on GitHub"
→ Documentation repos, SDKs, examples. Never the actual infrastructure code.
5
"Anyone can build on it"
→ Anyone can call the API. No one can verify, audit, or run their own instance.

When Spec ≠ Implementation

Even where specifications exist, they create a false sense of transparency:

◉ What Specs Don't Tell You

  • What data is actually logged
  • What metadata is retained
  • What side-channel leaks exist
  • What third parties have access
  • What the actual error handling does
  • Whether consent is actually enforced

◉ Known Spec-Reality Gaps

  • Aadhaar: "minimal data" spec vs. massive metadata collection
  • UPI: documented limits vs. actual transaction surveillance
  • ABDM: consent spec vs. reported unauthorized access
  • All systems: documented security vs. 210+ breaches
The Verification Problem

Without source code, there is no way to verify that the implementation matches the specification. A spec says "we do X." The code might do Y. Trust without verification is not transparency, it's faith.

The Export Paradox

MOSIP (Modular Open Source Identity Platform) is marketed as proof of openness:

◉ What's Open (for export)

  • Registration UI workflows
  • API middleware layer
  • Data schemas
  • Country customization hooks

◉ What's Proprietary (always)

  • Biometric matching (ABIS)
  • Fingerprint/iris/face recognition
  • De-duplication engine
  • The intelligence that matters
The Paradox

MOSIP is open source for export. Morocco, Philippines, Nigeria. All get the "open" framework. Meanwhile, Aadhaar, 17 years, 1.4 billion Indians, remains completely closed. Open for foreigners. Closed to citizens.

The Business Model

MOSIP is a sales funnel for biometric SDK vendors.

The open source wrapper (registration UI, workflows, APIs) creates adoption. Governments think they're avoiding vendor lock-in.

But identity verification requires ABIS, Automated Biometric Identification System. ABIS is proprietary. You must buy from TECH5, Idemia, NEC, or similar vendors.

The "open" part is the funnel. The closed part is where the money flows. This isn't a public good, it's a go-to-market strategy using open-washing.

The Government's Own Policy

Policy on Adoption of Open Source Software for Government of India (2015)

MeitY's official policy makes it "mandatory to formally adopt and use Open Source Software in government organisations, as a preferred option" compared to closed-source alternatives.

The policy requires:

  • All tech suppliers must submit bids with OSS options
  • Government must compare OSS alternatives for all procurements
  • Systems must use open standards for interoperability
  • Well-documented open APIs for migration and exit

→ MeitY Policy Document

The Contradiction

The government's own policy mandates OSS as the "preferred option." Yet every core India Stack component, Aadhaar, UPI, ABDM, DigiLocker, is closed source.

The policy exists. Compliance does not.

The Section 8 Bypass

The mechanism for evading policy: Section 8 companies (non-profits under Companies Act). These structures bypass government procurement rules, RTI, and OSS mandates.

  • NPCI (runs UPI): Section 8 company- not a government entity. RTI-exempt. OSS policy doesn't apply.
  • CoWIN: Funded by UNDP, developed by private companies. Not subject to government OSS procurement.
  • ONDC: Section 8 company, no writ petition possible, no RTI.
  • Digi Yatra Foundation: Private foundation (AAI 26% + 5 private airports).

And the enforcer violates its own rules: Aadhaar/UIDAI falls under MeitY, the same ministry that issued the OSS policy. MeitY is violating its own mandate.

Security Through Obscurity

Kerckhoffs's Principle (1883)

"A cryptosystem should be secure even if everything about the system, except the key, is public knowledge."

◉ Open Systems Track Record

  • Signal Protocol: Zero breaches
  • Linux Kernel: Powers 90% of cloud
  • Bitcoin: $800B+ secured 15 years

◉ India Stack Track Record

  • Aadhaar: 210+ documented breaches
  • UPI: 40+ vulnerabilities (2019)
  • ABDM: 31M records leaked (2024)
Assessment
India Stack is not observable. Therefore, it is not controllable by citizens.

OpenAPI docs are not source code. Specifications are not implementations. Marketing is not transparency.
