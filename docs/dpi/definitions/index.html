DPI Definitions: Structural Analysis

Every major DPI definition uses vocabulary that sounds like requirements, "open," "inclusive," "accountable," "public benefit." None define these terms operationally.

The promoters wrote the definitions.

The definitions exclude nothing the promoters built.

The same pattern applies when AI systems are promoted as Digital Public Infrastructure. Terms such as "safe," "aligned," and "responsible" read like requirements but lack operational definitions.

PATTERN IN VOCABULARY AND EFFECT

Words that sound like requirements, open, inclusive, accountable, interoperable, public benefit, appear repeatedly without operational definitions.

OPERATIONAL MAPPING

The corrigibility tests show what those words would require in practice:

  • open → CODE (source + reproducibility)
  • inclusive → EXIT (safe refusal)
  • accountable → AUDIT (permissionless verification)
  • interoperable → FORK (replaceability)

The Exposure Surfaces converts these implications into minimal, verifiable artifacts.

PUTTASWAMY INVARIANT

Consent requires the possibility of refusal without material harm, directly mapping to the EXIT test.

Every definition below treats DPI as infrastructure, roads for data traffic.

The corrigibility framework treats DPI as ecosystem, requiring stewardship, balance, and feedback.

Infrastructure metrics: scale, speed, volume. Ecosystem metrics: recourse, correction, survival.

This is why the same systems can be called "successful" by promoters and "lethal" by those excluded.

This mismatch explains why systems optimized for scale can remain structurally incapable of correction.

Why These Definitions Are Structurally Empty

1. Aspirational claims without verification

Terms such as "should be" or "for the public benefit" are preferences unless accompanied by verifiable rules and enforcement mechanisms.

2. Standards ≠ Source

Proprietary code can implement open standards.

3. Monopoly as dependency

Interoperability with a monopoly is dependency.

4. Legal frameworks do not guarantee protection

Every system operates under law; law alone does not prevent enclosure.

Ostrom: Disputes must be resolved locally. DPI centralizes redress.

5. No exit requirement

"Inclusive" often becomes compulsory. Without a right to refuse, inclusion is effectively compulsion.

Ostrom: Sanctions must be graduated. DPI delivers binary punishment: match or exclusion.

CONSTITUTIONAL FRAMEWORK

Right to Privacy: Constitutional Invariants

India's Right to Privacy judgment (2017), a globally significant constitutional ruling, identifies conditions under which judicial review of digital systems remains possible.

Invariant 1, Consent requires refusal

Refusal must not cause material harm. EXIT

Invariant 2, Legality requires knowable execution

Power must be contestable in operation. CODE AUDIT

Invariant 3, Proportionality presumes alternatives

Necessity collapses without replaceability. FORK

Invariant 4, Accountability requires independent review

Permission-based oversight is not oversight. AUDIT

Invariant 5, Power must remain reversible

Correction must occur before harm becomes irreversible. EXIT GOVERN FORK

Each invariant maps directly to a corrigibility condition.
