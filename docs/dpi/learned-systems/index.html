Annex: Learned Systems and Corrigibility

Governance for stochastic systems.

This annex explains how the corrigibility framework applies to learned or probabilistic systems, including contemporary AI models.

The five corrigibility tests remain invariant.

Same standard. Different verification method.

Corrigibility in deterministic systems concerns inspecting execution.

Corrigibility in learned systems concerns bounding behavior.

Learned systems require a Behavioral Specification, runtime disclosure, and drift telemetry in the Exposure Surfaces.

This annex does not introduce a weaker standard. It clarifies why certain learned systems may be structurally incapable of meeting the existing standard.

Scope of This Annex

This covers systems that learn behavior from data instead of following hardcoded rules:

  • Machine learning models
  • Large language models
  • Algorithmic decision systems trained on historical data
  • Systems whose outputs are probabilistic rather than deterministic

Systems that cannot satisfy all five tests do not meet the structural conditions required for designation as Digital Public Infrastructure.

The Deterministic Assumption

The five corrigibility tests implicitly assume deterministic systems.

Comparison of assumptions between deterministic and learned systems
Assumption Deterministic Systems Learned Systems
Predictability Same input produces same output Same input may produce different outputs
Source of behavior Source code Training data and objectives
Inspection Read the code Weights and representations are opaque
Reproduction Exact binary reproduction Statistical approximation only
Modification Change code and predict effect Change training with uncertain effects

Learned systems break every one of these assumptions.

These differences do not invalidate the framework. They identify where additional disclosure, monitoring, and governance mechanisms are required for learned systems to remain corrigible over time.

What Is "Source" in a Learned System

For traditional software:

source code → compiler → binary → behavior

For learned systems, behavior is distributed across layers:

Training data corpus
Data filtering and preprocessing decisions
Model architecture and hyperparameters
Random initialization and hardware effects
Training objectives and loss functions
Reinforcement learning from human feedback
System prompts and runtime constraints
Inference code

Inference code alone does not determine behavior. Publishing it does not satisfy the CODE test.

For learned systems, source transparency means transparency across the training pipeline.

The Five Tests Remain Invariant

The corrigibility framework continues to evaluate the same five power relationships.

How each test applies to deterministic vs learned systems
Test Deterministic Verification Learned System Verification
EXIT Can refuse the system Disclosure + non-AI alternative + query transparency
CODE Source code determines behavior Layered transparency across training pipeline
AUDIT Inspect logic and outputs Statistical bounds + continuous monitoring
GOVERN Maintainer and RFC process Governance over objectives and value tradeoffs
FORK Copy code and deploy Resource forkability including compute and data

No new test is introduced. No existing test is relaxed.

EXIT, EXTENDED

Disclosure and Alternative

Learned systems are often embedded invisibly. Users may not know when an AI system is mediating access to services, recommendations, or decisions.

EXIT for learned systems requires:

  • Mandatory disclosure that AI is involved
  • Disclosure of what role the system plays
  • A non-AI alternative without penalty
  • Query-level transparency including model version and policy version

If refusal results in degraded service, exclusion, or delay, EXIT fails.

CODE, EXTENDED

Layered Transparency

In learned systems, behavior is determined primarily by training.

CODE requires disclosure of:

  • Training data sources and provenance
  • Data inclusion and exclusion criteria
  • Training objectives and loss functions
  • Hyperparameters and architecture choices
  • RLHF or preference optimization processes
  • System prompts and safety constraints
  • Version history of each layer

Publishing model weights without disclosure of training data, training objectives, and optimization processes provides insufficient legibility for the purposes of structural accountability.

AUDIT, EXTENDED

Statistical Verification

Learned systems cannot be fully audited through inspection. Audit therefore shifts from logical verification to statistical verification.

Audit approach comparison
Deterministic Audit Learned System Audit
Does this rule fire? Does output distribution remain within bounds?
Is there a bug? What is the error rate under distribution D?
One-time review Continuous monitoring
Deterministic claims Confidence-bounded claims

Audit for learned systems requires:

  • Behavioral specifications
  • Defined acceptable error and exclusion bounds
  • Continuous measurement against those bounds
  • Disclosure of model updates and retraining events
  • Independent audit access without permission

Benchmark passing is not audit.

For learned systems, auditability is satisfied through confidence-bounded claims supported by continuous measurement. Point-in-time certification or benchmark performance does not constitute independent verification of deployed behavior.

GOVERN, EXTENDED

Objective Governance

In deterministic systems, governance operates through code review and maintainership. In learned systems, governance operates through objectives.

Governance questions include:

  • What is the system optimizing for?
  • What outcomes are prioritized?
  • What tradeoffs are accepted?
  • What constitutes harm or error?
  • Who defines acceptable failure rates?

These are normative decisions expressed through technical parameters.

For learned systems, GOVERN requires:

  • Transparency of objectives and value tradeoffs
  • Binding input from affected populations
  • Public processes for changing objectives
  • Disclosure of who sets preferences and how

Grievance mechanisms after harm do not constitute governance.

FORK, EXTENDED

Resource Forkability

Forking a learned system is constrained by resources, not just permission.

Resource barriers to forking learned systems
Layer Typical Cost Barrier
Inference code Minimal None
Running open weights Low Minor
Fine-tuning Moderate Economic
Retraining from scratch Very high Structural

True forkability requires:

  • Access to sufficient compute
  • Access to recreatable or alternative data
  • Disclosure of training recipes
  • Economic feasibility for non-incumbents

Legal permission to fork without access to compute or data is meaningless. This is an infrastructure question, not a licensing question.

Training Data as a Commons

Training data is collectively generated across societies, while model development and control remain concentrated among a limited number of actors. This asymmetry transfers value without corresponding governance rights.

Current systems derive legitimacy from collective production without adopting the governance structures that commons require.

Commons governance requires:

  • Clear boundaries: who can use what
  • Monitoring accountable to contributors
  • Graduated sanctions for misuse
  • Dispute resolution that works
  • Rule-making by those whose data it is

Current AI systems fail all five.

No governance of the data commons, no public infrastructure.

Probabilistic Power

Traditional systems say: "If X, then Y." You can read the rule. You can contest it.

Learned systems say: "Probably Y." You cannot read why. You can only observe patterns.

Power now looks like:

  • Patterns learned from historical data, including historical bias
  • Weights you cannot read
  • Outcomes that vary per person
  • Objectives set by operators, not users

This represents a form of governance mediated through statistical inference rather than explicit rules.

The question changes:

Before: "Is this rule correct?"

Now: "Is this error rate acceptable, and who decides?"

The Correction Gap

Traditional fix:

bug reported → find the line → patch it

AI fix:

problem reported → which training data? → which objective? → retrain everything?

Retraining is not patching. You might need new data, new objectives, new compute. Months, not hours.

The framework still applies. But the gap limits where AI can go.

Where correction cycles exceed acceptable time horizons for the function being performed, deployment scope must be correspondingly constrained.

If correction cannot occur within the timeframe required by the function, deployment is unsuitable for that function.

REQUIRED ARTIFACT: CORRECTION PATHWAY

Every learned-system submission must include a short, testable "Correction Pathway" document that states:

  1. Detection triggers: telemetry/drift thresholds with quantitative thresholds and false-positive/false-negative tradeoffs.
  2. Fallback delivery method: how affected users continue to receive service while correction occurs.
  3. Remediation window: maximum time for restoration, with confidence bounds on expected recovery.
  4. Escrow triggers: what public artifacts unlock if remediation fails.

This Correction Pathway is a checklist item in the Exposure Surfaces.

Implications for Designation and Deployment

Learned systems may be evaluated under the same corrigibility framework applied to deterministic digital public infrastructure. The five tests remain invariant; what differs is the architectural and evidentiary burden required to satisfy them.

Where a learned system cannot be safely refused without penalty, rendered legible across its training pipeline, independently audited through statistical verification, governed over its objectives by affected populations, and credibly replaced within feasible resource constraints, it lacks the structural properties required for designation as Digital Public Infrastructure.

This assessment is structural rather than normative. It does not evaluate intent, innovation, or performance. It evaluates whether affected parties retain the capacity to detect error, contest outcomes, and trigger correction before harm becomes irreversible.

In practice, many contemporary learned systems exhibit correction cycles, resource concentration, and opacity that exceed acceptable thresholds for essential or consequential public functions. In such cases, deployment may remain appropriate for advisory, experimental, or non-essential use, but not for population-scale public authority.

Designation as public infrastructure is conditional on corrigibility being demonstrable in operation, not asserted in principle.

Same five tests. Same standard. No exceptions for AI.

What differs is the evidentiary and architectural burden required to satisfy them.

This annex does not weaken the framework. It identifies the conditions under which learned systems may satisfy it.

For learned systems to qualify as public infrastructure, they require disclosure mechanisms, governance arrangements, and resource accessibility that most current systems do not provide.
