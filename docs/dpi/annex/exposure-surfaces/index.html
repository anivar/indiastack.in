ANNEX

Corrigibility Exposure Surfaces

Structural locations where claims about a system can be independently challenged by affected parties.

Exposure surfaces reveal boundaries of visibility, contestability, and replaceability. They express structure rather than compliance. Each surface maps to a feedback entry point (cybernetics), a monitoring boundary (commons governance), or a replaceability threshold (FOSS).

WHY THIS ANNEX EXISTS

This annex exists to make corrigibility surfaces explicit. The core DPI framework defines what must be structurally possible for infrastructure to be public. This annex identifies where and how those possibilities surface in real systems. In the FOSS tradition, observability concerns whether execution can be inspected and replaced. Corrigibility extends further: it asks where affected parties can introduce feedback, contestation, and correction across exit, audit, governance, and replacement boundaries. Exposure surfaces do not assert compliance, safety, or legitimacy. They describe structural locations where claims about a system can be independently challenged, or where such challenge is structurally blocked. Where no exposure surface exists, correction is impossible regardless of intent. This annex therefore supports analysis, not certification; diagnosis, not endorsement; and correction, not trust.

STRUCTURAL DEFINITION

Exposure surfaces are structural locations where claims about a system can be independently challenged and acted upon by affected parties.

They describe where correction can enter a system. Each surface corresponds to one or more of these functional properties:

  • Feedback entry: a point where affected parties can send corrective signals that change system behavior.
  • Monitoring boundary: a place where reciprocal observation enables collective governance.
  • Replaceability threshold: a boundary where alternative implementations can emerge without incumbent permission.

These properties map to cybernetics (feedback), commons governance (monitoring), and FOSS (replaceability) respectively.

EXIT

Refusal Surface

A public map of alternative, non-digital pathways for each essential service endpoint.

  • Public statement of non-digital alternatives for each essential service endpoint.
  • Published fallback procedures and access methods (phone lines, physical office locations, alternate identity paths).
  • Disclosure of service mediation pathways and consequences of refusal, with practical guidance on continuity of access.

Exit surfaces prioritize user continuity: refusal and fallback deliver structurally equivalent service levels while correction proceeds.

CODE

Execution Visibility Surface

A structural map of where execution logic becomes observable and reproducible.

  • Source repository links and OSI-compatible license declarations.
  • Build manifest, deterministic build scripts, and signed reproducible-build checksums; or a link that ties deployed binary to a published reproducible build run.
  • For learned systems: training pipeline summary (data classes, filtering rules, optimization objective), training recipe reference, and provenance pointers for major datasets.
  • Behavioral Legibility Layer:
  • Behavioral Specification Language (BSL): machine-readable statement of input classes, expected outputs, acceptable error bounds stratified by relevant groups, confidence intervals, and test distributions.
  • Runtime configuration disclosure: deployed model/version identifier, versioned prompt templates or runtime policy snippets, guardrail/filter descriptions, and change history.

Code surfaces prioritize legibility and reproducibility so that production behavior links back to inspectable artifacts.

AUDIT

Verification Surface

A structural map of independent observation points for system behavior.

  • Telemetry schema and public dashboard with stratified aggregate metrics (error and rejection rates by group and region).
  • Description of the test harness and sample black-box test cases used for verification.
  • Named independent auditors or contact points for independent testing.
  • Drift and detection telemetry: documented drift metrics, alert thresholds, and time-stamped summaries of recent drift events and remediation status.

Audit surfaces operate independently of operator discretion, enabling continuous verification and public evaluation.

Where audit depends on operator-granted access, trust substitutes for structure.

GOVERN

Governance Surface

A structural map of where rules form and change with binding effect.

  • Public policy-as-code repository or published changelog documenting eligibility and operational rules in machine-readable form.
  • Record of affected-population consultation (RFCs, public comment summaries, minutes) with evidence of how input shaped decisions.
  • Decision authority: which bodies hold binding decision power, which roles can trigger policy changes, and mechanisms for affected-party representation.

Govern surfaces expose where collective choice arrangements and sanctioning rules are structured.

Constitutional review presupposes these surfaces. Where they are absent, proportionality analysis cannot function.

FORK

Replacement Surface

A structural map of legal, technical, and economic conditions that permit replacement.

  • License permitting redistribution and fork.
  • Declaration of runtime dependencies required for operation (proprietary services, vendor SDKs, hardware keys).
  • Statement of practical barriers to forkability and mitigations.
  • For learned components: a resource-access statement describing compute, dataset, and knowledge barriers and possible mitigations (public compute grants, dataset mirrors, training recipes).

Fork surfaces express whether replacement is feasible in law, technology, and economics.

Economic infeasibility is structural infeasibility. Where replacement requires resources only incumbents control, permission is granted in form but withheld in fact.

LEARNED SYSTEMS: CORRECTION PATHWAY

Learned systems exercise statistical governance. Corrigibility therefore operates through bounded behavior, continuous observation, and resource-level replaceability. A Correction Pathway for learned systems maps:

  • Detection triggers: drift thresholds and detection telemetry with quantitative sensitivity and specificity tradeoffs.
  • Fallback delivery: mechanisms by which affected users receive continuing service while correction is implemented.
  • Remediation window: maximum timeframes for restoration, with confidence bounds on recovery expectations.
  • Escrow triggers: artifacts and disclosures that become accessible if remediation exceeds the remediation window (for example, model snapshots, training recipe logs, or external audit activation).

Correction pathways align operational change with temporal safeguards and public observability.

Service continuity invariant: correction must not require service interruption for affected users. Fallback delivery must be structurally guaranteed.

TRUST DEPENDENCIES

Where system behavior depends on operator discretion, these dependencies appear explicitly and structurally:

  • Permission-based audit access and the scope of required permissions.
  • Sampled or aggregated metrics and sampling methodology.
  • Contractual testing limits and terms that shape third-party testing.
  • Proprietary dependencies and single-vendor chokepoints.
  • Resource concentration: compute, curated datasets, and specialist knowledge.

Trust dependency fields identify where correction routes through human trust rather than structural guarantees.

Trust is not corrigibility. Where trust is required, correction is contingent on continued goodwill.

Structural Notes

  • Each surface points to a reproducible artifact, a public interface, a measurable boundary, or an explicit opacity marker.
  • Omitted surfaces function as explicit exposure choices and signal where independent challenge requires additional effort.
  • Partial openness produces partial exposure; corrigibility operates at the narrowest exposed layer.
  • The framework defines structural preconditions for corrigibility and provides vocabulary for analysis. It supplies the topology that enables independent challenge and public contestation.

TWO-MANIFEST ARCHITECTURE

The corrigibility framework does not prescribe instruments. It evaluates structure. Two complementary schemas separate operator facts from auditor judgments. Operators cannot grade their own homework.

INFRASTRUCTURE.JSON

Operator Manifest

Published by system operators. Declares operational facts: dependencies, access channels, governance model. Does not assert compliance.

Analogy: package.json

CORRIGIBILITY.JSON

Auditor Assessment

Authored by auditors, not operators. Evaluates structural correction capacity against the five tests. Contains pass/fail judgments with evidence.

Analogy: security audit report

These schemas are NOT:

  • Compliance attestations or self-certifications
  • Safety declarations
  • The only way to expose corrigibility surfaces

They are: One possible disclosure format. The framework is upstream of these tools.

Infrastructure Manifest (v1.4)

Location: /.well-known/infrastructure.json

Machine-readability enables automated discovery. Boolean fields force confession. Named arrays force topology mapping. Each section is an evidence surface for the five tests.

SCHEMA STRUCTURE

Both schemas use JSON Schema Draft 2020-12. Validate against: infrastructure.json and corrigibility.json

INFRASTRUCTURE.JSON v1.4

  • meta: system_id, deployment, lifecycle
  • operations: status_page, slos (uptime_target, dispute_resolution)
  • governance: model, escalation_paths[], public_participation
  • components: repositories[], closed_components[]
  • learned_components: array of {name, role, when_uncertain, model_cards[]}
  • access: endpoints[], offline_channels[], offline_equivalent

CORRIGIBILITY.JSON v1.4

  • target: matches infrastructure system_id
  • assessed_by: auditor organization
  • assessed_at: required date (prevents stale assessments)
  • tests: exit, code, audit, govern, fork (each with evidence[])
  • verdict: corrigible, tests_passed, summary

THE TRAP: OPERATOR vs AUDITOR

Operator claims offline_equivalent: true → Auditor measures exit.penalty_ratio: 50 → Proof of Lie

Operator claims dispute_resolution: "72h" → Auditor records observed_fix_time: "3 months" → Proof of Failure

Operator claims escalation.binding: true → Auditor finds govern.user_power: "advisory" → Proof of Theater

PRACTICAL GUIDANCE

Primary purpose: Map where independent actors can observe, test, and challenge system behavior. Supply machine-readable pointers to the artifacts that enable independent scrutiny.

Who uses it
  • Civil society and journalists for triage
  • Researchers for automated discovery
  • Auditors preparing test harnesses
  • Policy drafters writing procurement clauses
  • Automated agents indexing exposure surfaces
Intended uses
  • Generate human review tasks
  • Produce prioritized verification checklists
  • Feed discovery services
  • Identify missing surfaces quickly
Out-of-scope uses
  • Automated certification or enforcement decisions
  • Single-field pass/fail judgments
  • Claims that manifest alone establishes corrigibility
Operational workflow
  • Publish: Operator publishes manifest at stable URL
  • Index: Agents parse manifest structure
  • Triage: Agents compute asymmetry metrics
  • Test: Humans verify linked artifacts
  • Update: Operators update change_log

Corrigibility becomes observable where exposure maps supply clear interfaces for challenge, correction, and replacement.
