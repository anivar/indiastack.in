Evaluation Procedure

Methods

This site evaluates digital systems using structural tests, not outcomes. This page explains why, and how.

SCOPE

This page defines how systems are evaluated. It does not argue what systems should exist.

The Problem with Outcomes

At population scale, digital systems cease to be tools. They become control systems, closed loops that act on populations, measure responses, and adjust behavior.

When participation is compulsory or refusal carries material cost:

  • Adoption measures dependence, not preference
  • Usage measures compliance, not consent
  • Scale measures reach, not legitimacy
  • Satisfaction surveys measure adaptation to constraint, not free choice

The signal collapses. Coerced data cannot distinguish success from legitimacy. Outcome metrics are therefore excluded from evaluation on this site.

Three Analytical Lenses

This site applies three independent frameworks. Each emerged from a different discipline. Each addresses a different domain. They converge diagnostically.

These frameworks do not reinforce each other rhetorically. They converge on the same structural requirements because those requirements are necessary for legitimate shared systems.

The Core Criterion
Corrigibility
/kɒˈrɪdʒɪbɪlɪti/
The structural capacity of those affected by a system to detect error, apply pressure, and trigger correction, without suffering material harm.

A corrigible system can be observed (its rules are visible), tested (its claims can be verified), refused (exit does not destroy the person), shaped (affected populations have constitutive input), and replaced (alternatives can exist).

Without corrigibility, systems mistake compliance for consent, treat silence as approval, suppress harm signals as noise, and escalate control when things break. Error costs land on users; the system stays safe for operators.

Five Structural Tests

A system is classified as public only if all five are satisfied:

TEST 01
Safe Refusal →
Refusal of digital systems must be possible while retaining access to non-digital government services.
Consent given under duress is not consent. If declining digital means losing government services entirely, the "choice" is meaningless.
TEST 02
Legible Execution →
Rules that materially affect people must be inspectable in their executable form.
The specification is not the system. The API is not the algorithm. Only executable code reveals actual behavior.
TEST 03
Independent Verifiability →
Claims about system behavior must be continuously testable by independent, adversarial actors without prior permission.
Trust that cannot be verified is obedience. Internal audits are structurally compromised.
TEST 04
Constitutive Input →
Affected populations must have rule-making input over system design, not merely grievance mechanisms after harm.
Grievance redressal is remedial. Constitutive input shapes rules before harm. These are structurally different.
TEST 05
Credible Replacement →
It must be technically and legally possible to collectively replace or replicate the system without incumbent permission.
The possibility of replacement disciplines power even when unused. When alternatives are forbidden, the system becomes monopoly.
Binary Tests
Failure of any one test is fatal.
Partial compliance does not count. These are necessary conditions, not a scoring rubric.

A system that is highly transparent but offers no exit is still incorrigible. A system that permits exit but cannot be audited is still incorrigible. Partial feedback is not feedback, it is noise that legitimates while failing to constrain.

What This Site Does
DOES NOT
  • Argue intent or motivation
  • Debate efficiency or innovation
  • Weigh benefits against harms
  • Propose alternative systems
  • Claim outcomes are bad
DOES
  • Apply structural tests consistently
  • Classify systems as corrigible or not
  • Document evidence for those tests
  • Refuse to treat scale as legitimacy
  • Maintain procedural neutrality

The site's position is not "India Stack is bad." It is: India Stack fails structural tests required for public legitimacy, therefore outcome claims are not evidence, therefore the system cannot be defended on the grounds usually offered.

A digital system is legitimate only if it is structurally corrigible by those it affects.

Absent that: efficiency is compression of dissent, stability is delayed failure, and "public" is a marketing term.

Readers are encouraged to apply this method to any digital system, not only those discussed here.
